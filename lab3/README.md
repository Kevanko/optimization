## Лабораторная работа 3 — «Диспетчер–вычислительный центр»

- **Суть задачи**: есть вычислительный центр (ВЦ) с \(n\) одинаковыми ЭМ и диспетчер, у которого в очереди задачи разных рангов. Между ВЦ и диспетчером формулируется матричная антагонистическая игра: ВЦ выбирает, сколько машин держать включёнными, диспетчер — сколько задач подать. Платёж — суммарные эксплуатационные затраты (работающие, простаивающие и ожидающие задачи).
- **Что делает программа**: по заданным \(n, c_1, c_2, c_3, \varepsilon\) строит матрицу платежей \(C\) по формуле из задания и с помощью итерационного метода Брауна находит приближённое значение цены игры и оптимальные смешанные стратегии ВЦ и диспетчера.

### Реализация

- `matrix.py` — формирование матрицы платежей \(C\) размера \((n+1)\times(n+1)\) по формулам из примера работы (матрица для \(n=10, c_1=1, c_2=2, c_3=3\) совпадает с приведённой в задании).
- `brown.py` — метод Брауна (фактически фиктивная игра): попеременное построение наилучших ответов двух игроков с учётом накопленных выигрышей, отдельный учёт числа ходов каждого игрока, критерий остановки по зазору \(\beta-\alpha < \varepsilon\).
- `main.py` — разбор параметров из командной строки или файла, вывод матрицы, числа итераций, зазора \(\beta-\alpha\), оценочного значения игры \((\alpha+\beta)/2\) и смешанных стратегий.
- `plot.py` — замер времени работы метода Брауна при разных \(n\) и построение графика зависимости времени от числа машин.

Запуск (пример):

- `make run` — решение игры для выбранных параметров (по умолчанию берутся значения варианта).
- `make example` — пример из методички: \(n=10, c_1=1, c_2=2, c_3=3, \varepsilon=0.01\).
- `make plot` — построение графика времени работы `plots/runtime_vs_n.png`.

### График времени работы и выводы

- **График**: по оси X — число машин \(n\), по оси Y — время работы алгоритма (секунды). Кривая монотонно растёт: при увеличении \(n\) размер матрицы и длина итераций возрастают, а также обычно требуется больше шагов до выполнения условия \(\beta-\alpha<\varepsilon\).
- **Интерпретация**: при малых \(n\) метод Брауна работает очень быстро, при \(n \approx 30\!-\!50\) время становится заметным, но остаётся приемлемым для лабораторных масштабов. Это согласуется с теорией: метод является итерационным, но для матриц умеренного размера и не слишком строгого \(\varepsilon\) даёт хорошее приближение за разумное число шагов.

### Ответы на вопросы / комментарии по сути лабы

- **Что оптимизируется**: ВЦ стремится минимизировать средние затраты, диспетчер — максимизировать их; найденные смешанные стратегии дают компромиссный, «игровой» режим работы системы.
- **Функционирование ВЦ и диспетчера при найденных стратегиях**:
  - ВЦ не держит фиксированное число машин включёнными, а **случайным образом** (с заданными вероятностями) выбирает один из уровней мощности. Это позволяет сгладить риск «провальных» ситуаций, когда диспетчер всегда подстраивается под один и тот же режим.
  - Диспетчер, зная стратегию ВЦ, также распределяет поток задач вероятностно, избегая стратегий, которые дают ВЦ слишком большую выгоду.
- **Качество решения**: малый зазор \(\beta-\alpha\) (по сравнению с масштабом платежей) говорит о том, что найденные стратегии близки к седловой точке. В примерах зазор порядка \(\varepsilon\), что соответствует требованиям.
- **Моё мнение по модели**: модель удобна для учебных целей — наглядно показывает, как с помощью теории игр можно формализовать взаимодействие подсистем (здесь — ВЦ и диспетчер) и обосновать вероятностные решения вместо «жёстких» правил. Для реальных систем важны более детальные модели стоимости и ограничений, но сам подход (формирование матрицы и поиск смешанных стратегий) остаётся полезным.
